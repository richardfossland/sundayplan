# @sundayplan/db

Supabase migrations + generated types for SundayPlan.

## Migrations

Located in `migrations/`, numbered and applied in order. The CLI reads them via:

```bash
# Local dev (Docker required):
supabase start
supabase db reset            # reapplies all migrations from scratch

# Push to a remote project:
supabase db push
```

### Current migrations

- `0001_tenancy.sql` — Church (tenant root), user profile, ChurchMember,
  RLS helper functions (`is_member_of`, `is_planner_of`).
- `0002_core.sql` — full domain model: members, teams, roles, availability,
  services, songs, assignments, magic-link tokens, comms logs. RLS policies
  on every table.

## Generated types

After applying migrations, regenerate TypeScript types for the SDK:

```bash
pnpm db:types     # → types.generated.ts
```

This is git-ignored — it's regenerated by CI on every push.

## Seed data

`seed/` holds Norwegian-flavoured sample data for local development:
"Alta Frikirke", 3 teams, 15 members, 4 services planned. To be filled in
Phase 1.2.

## RLS testing

Test cases live in `tests/` once Phase 1.2 is complete. Each scenario
exercises a specific policy: planner can see all members, volunteer
magic-link cannot see other members' assignments, etc.
